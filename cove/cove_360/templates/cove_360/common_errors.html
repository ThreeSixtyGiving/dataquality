{% trans "Invalid 'uri' found" %}

{% blocktrans %}This message means you have a problem, usually with a website address in your data.
This error is often caused by the following problems:{% endblocktrans %}

  • {% blocktrans %}The uri does not start the address with 'http://' e.g. www.example.com is not a 'uri' but http://www.example.com is. All uri should start with either http:// or https://{% endblocktrans %}
  • {% blocktrans %}There are spaces at the start or end of the address which need to be removed. Be aware that these spaces can often be invisible.{% endblocktrans %}
  • {% blocktrans %}There is more than one address in the same field e.g. "http://www.example.com and http://www.example2.com"{% endblocktrans %}

{% blocktrans %}What is a URI? Short for “Uniform Resource Identifier” it is a sequence of characters that distinguishes one resource from another. A URL — short for “Uniform Resource Locator” — is a specific type of URI.{% endblocktrans %}

{% blocktrans %}When the 360Giving Data Standard requires data to be provided in the format of a URI, this will mean you will very likely be providing a URL — or website address.{% endblocktrans %}

{% trans "Date is not in the correct format" %}

{% blocktrans %}Dates should be in the formats defined by the 360Giving Data Standard.{% endblocktrans %}

{% blocktrans %}All Last Modified fields should be in datetime format and all other dates, such as Award Dates and Planned Dates, should be in the YYYY-MM-DD format. However, Award and Planned Dates can also be provided in datetime format.{% endblocktrans %}

{% blocktrans %}YYYY-MM-DD format used for full dates{% endblocktrans %}

{% blocktrans %}Dates should be in the form YYYY-MM-DD where:{% endblocktrans %}

  • {% blocktrans %}YYYY represents a year e.g. 2022{% endblocktrans %}
  • {% blocktrans %}MM represents a 2 digit month of the year e.g. 01 for January{% endblocktrans %}
  • {% blocktrans %}DD represents a 2 digit day of the month e.g. 01 for the 1st{% endblocktrans %}

{% blocktrans %}Example: 2022-01-01{% endblocktrans %}

{% blocktrans %}Common date errors include{% endblocktrans %}

  • {% blocktrans %}providing non-existent dates e.g. 2022-02-31 (31st February 2022){% endblocktrans %}
  • {% blocktrans %}non-date content in a date field (by accident){% endblocktrans %}

{% blocktrans %}Datetimes format used for Last Modified fields{% endblocktrans %}

{% blocktrans %}All Last Modified fields should be in datetime format, which is a very specific form of a date and time that is specified by the 360Giving Data Standard.{% endblocktrans %}

{% blocktrans %}A datetime should be in the form of YYYY-MM-DDThh:mm:ss±hh:mm where:{% endblocktrans %}

  • {% blocktrans %}YYYY represent a year e.g. 2022{% endblocktrans %}
  • {% blocktrans %}MM represents a 2 digit month of the year e.g. 01 for January{% endblocktrans %}
  • {% blocktrans %}DD represents a 2 digit day of the month e.g. 01 for the 1st{% endblocktrans %}
  • {% blocktrans %}T is simply the letter 'T'{% endblocktrans %}
  • {% blocktrans %}hh is a 2 digit representation of an hour e.g. 01 for 1AM, 13 for 1PM{% endblocktrans %}
  • {% blocktrans %}mm is a 2 digit representation of minutes in an hour e.g. 05 for 5 past{% endblocktrans %}
  • {% blocktrans %}ss is a 2 digit representation of seconds{% endblocktrans %}
  • {% blocktrans %}+hh:mm or -hh:mm is a timezone offset from UTC. For most UK data producers this can simply be replaced with a 'Z'{% endblocktrans %}

{% blocktrans %}Example: 2022-01-01T12:00:00Z is 12 noon on the 1st January 2022 in UTC time{% endblocktrans %}

{% blocktrans %}You can find further guidance here about date and datetime fields in the 360Giving data, as well as formatting tips when publishing using spreadsheets.{% endblocktrans %}
